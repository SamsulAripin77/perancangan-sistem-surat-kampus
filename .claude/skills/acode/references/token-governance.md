# Token Governance

The primary token-saving mechanism is portable project memory and selective context loading, not a precise prediction of API billing.

## Optimize the real causes

1. Reconstructing project state after every chat switch.
2. Re-reading the full development plan or archive.
3. Repeatedly sending unchanged source files or conversation context.
4. Reading full files when one symbol or line range is enough.
5. Dumping verbose builds, logs, database rows, or generated assets.
6. Broad repository exploration without a dependency reason.
7. Continuing failed investigation paths after evidence is sufficient.

## Startup context order

1. `PROJECT-STATE.json`
2. `WORK-QUEUE.md`
3. selected `CURRENT-TASK.md`
4. directly referenced decision/module sections
5. targeted source and tests

Archive and complete development-plan reads are exceptions, not defaults.

## Estimates

Mode and estimates describe planned behavior and visible context pressure. They cannot fully include system prompts, tool schemas, cached context, repeated model calls, retries, reasoning tokens, or internal agent turns.

Use actual API/runtime telemetry when available. Otherwise label estimates as planning-only ranges, never as billing forecasts.
